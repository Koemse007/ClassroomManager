# 1ï¸âƒ£ SYSTEM OVERVIEW

## 1.1 Problem Statement

The Classroom Management System addresses the need for digital collaboration between teachers and students:

**Current Challenges:**
- Teachers manually manage assignments and grades
- Students cannot easily track deadlines or submission status
- No centralized platform for group-based learning
- Lack of analytics on student performance
- Manual file management and storage

**Proposed Solution:**
- Centralized platform for group management
- Automated task assignment with deadlines
- Digital submission and grading workflow
- Real-time performance analytics
- Secure file storage

## 1.2 System Scope

**In Scope:**
- User authentication (teachers & students)
- Group/class creation and management
- Task assignment with deadlines
- Student submissions with file support
- Teacher grading and feedback
- Performance analytics and reporting

**Out of Scope:**
- Video conferencing
- Real-time chat
- Email notifications (future feature)
- Mobile app (future)
- LMS integration

## 1.3 Stakeholders

| Stakeholder | Role | Needs |
|-------------|------|-------|
| **Teachers** | Content creators | Easy task creation, grade management, analytics |
| **Students** | Content consumers | Task visibility, submission tools, feedback |
| **Administrators** | System managers | User management, system monitoring |
| **Parents** | Observers | Student progress tracking (future) |

## 2.3 Detailed Use Cases

### UC-001: Teacher Creates Group

```
Title: Create Classroom Group
Actor: Teacher
Precondition: Teacher is logged in

Main Flow:
  1. Teacher clicks "Create Group"
  2. System displays group creation form
  3. Teacher enters group name
  4. Teacher submits form
  5. System generates 6-character join code
  6. System stores group with teacher as owner
  7. System displays join code to teacher
  8. System confirms "Group created successfully"

Postcondition: Group exists and is ready for students to join
Alternative: 
  - 3a. Group name already exists â†’ Show error, prompt for new name
  - 3b. Name exceeds 100 characters â†’ Show error, allow edit
```

### UC-002: Student Joins Group

```
Title: Join Classroom Group
Actor: Student
Precondition: Student is logged in, has join code

Main Flow:
  1. Student clicks "Join Group"
  2. System displays join group dialog
  3. Student enters join code
  4. System validates code exists and is valid
  5. System adds student to group membership
  6. System confirms "You joined [Group Name]"
  7. System redirects to group dashboard

Postcondition: Student is member of group, can see tasks
Error Cases:
  - Invalid code â†’ Show error message
  - Already member â†’ Show info message
  - Code expired â†’ Show error with renewal option
```

### UC-003: Teacher Creates Task with Deadline

```
Title: Create Assignment Task
Actor: Teacher
Precondition: Teacher is logged in, group exists

Main Flow:
  1. Teacher selects group
  2. Teacher clicks "Create Task"
  3. System displays task creation form
  4. Teacher enters:
     - Title
     - Description
     - Due date/time
     - Optional file attachment
  5. Teacher submits form
  6. System validates deadline is in future
  7. System stores task with group_id
  8. System notifies students in group
  9. Confirms "Task created"

Postcondition: Task is visible to all students in group
Exceptions:
  - Past deadline â†’ Reject, show error
  - File > 10MB â†’ Reject, show error
```

### UC-004: Student Submits Task

```
Title: Submit Assignment
Actor: Student
Precondition: Student is in group, task is not deleted

Main Flow:
  1. Student navigates to group tasks
  2. Student clicks task
  3. System displays task details and submission form
  4. Student enters submission content (text and/or file)
  5. Student clicks submit
  6. System validates not empty
  7. System stores submission with current timestamp
  8. System marks task as "Submitted"
  9. System shows "Submission successful"

Postcondition: Submission recorded, visible to teacher
Variations:
  - Late submission â†’ Accept but flag as "Late"
  - File + Text â†’ Both accepted
```

### UC-005: Teacher Grades Submission

```
Title: Grade Student Submission
Actor: Teacher
Precondition: Submission exists and ungraded

Main Flow:
  1. Teacher navigates to "Pending Grades"
  2. System lists ungraded submissions
  3. Teacher clicks submission
  4. System displays:
     - Student name
     - Submission content/file
     - Due date
  5. Teacher enters:
     - Score (0-100)
     - Feedback text
  6. Teacher submits grade
  7. System stores grade with timestamp
  8. System notifies student
  9. Confirms "Grade recorded"

Postcondition: Grade visible to student
Constraints:
  - Score must be 0-100
  - Feedback optional but encouraged
```

From use cases, we identify these domain objects:

| Object | Attributes | Responsibilities |
|--------|-----------|-----------------|
| **User** | id, name, email, password, role | Authenticate, manage profile |
| **Group** | id, name, ownerId, joinCode | Manage members, create tasks |
| **GroupMember** | id, groupId, userId | Track membership |
| **Task** | id, groupId, title, description, dueDate, fileUrl | Manage deadline, accept submissions |
| **Submission** | id, taskId, studentId, content, fileUrl, submittedAt, score | Track submission, store feedback |
| **Analytics** | userId, groupId, statistics | Calculate performance metrics |

## 8.1 Coding Standards

```typescript
// âœ“ Good: Strong typing
interface User {
  id: string;
```

### 2. **Error Handling**
```typescript
// âœ“ Good: Explicit error handling
try {
  const user = await storage.getUserById(id);
  if (!user) {
    throw new Error("User not found");
  }
  return user;
} catch (error) {
  logger.error("getUserById failed:", error);
  throw error;
}

// âœ— Bad: Silent failures
const user = await storage.getUserById(id);
return user; // Could be undefined
```

## 8.3 Database Design Principles

### 1. **Normalization**
- Eliminate data redundancy
- Reduce update anomalies
- Maintain referential integrity

### 2. **Foreign Key Strategy**
```sql
-- SET NULL: Preserve history when reference deleted
FOREIGN KEY (ownerId) REFERENCES users(id) ON DELETE SET NULL

-- CASCADE: Remove related records (use sparingly)
FOREIGN KEY (groupId) REFERENCES groups(id) ON DELETE CASCADE
```

### 3. **Indexing Strategy**
- Index frequently queried columns
- Index foreign keys
- Index unique constraints
- Avoid over-indexing (slows writes)

## 8.4 Security Best Practices

### 1. **Authentication**
- Use JWT tokens with expiration
- Hash passwords with bcrypt
- Validate tokens on each request

### 2. **Authorization**
- Check user role before sensitive operations
- Verify ownership of resources
- Use middleware for enforcement

### 3. **Input Validation**
- Validate all user input
- Use parameterized queries (ORM)
- Sanitize file uploads

### 4. **File Security**
- Validate file types
- Limit file size
- Store outside web root
- Generate unique filenames

## 8.5 Performance Optimization

### 1. **Caching**
- Cache frequently accessed data
- Use TanStack Query for client caching
- Implement TTL for server cache

### 2. **Database Queries**
- Use indexes on hot columns
- Avoid N+1 queries
- Preload related data

### 3. **Frontend Optimization**
- Code splitting by route
- Lazy load components
- Memoize expensive computations
